# Configuration Management Plan

## 1. Purpose

This document defines the configuration management (CM) plan for RESONANCE, satisfying IEC 62304:2006+Amd1:2015 §8 (Software configuration management). It establishes which items are under configuration control, how they are versioned and identified, how changes are controlled, and how baselines are defined and maintained.

RESONANCE is classified as IEC 62304 Class A (RD-1.2). For Class A software, IEC 62304 requires:
- Identification of configuration items (§8.1.1)
- Change control (§8.1.2)
- Configuration status accounting (§8.1.3)

This plan documents existing practices (Git-based version control, Cargo.lock dependency pinning, sprint-based change process) and formalizes them for regulatory traceability.

### 2.3 Configuration Item Identification

Each configuration item is uniquely identified by:

1. **File path** -- relative to repository root (e.g., `src/blueprint/equations/determinism.rs`)
2. **Git commit hash** -- 40-character SHA-1 identifying the exact version (e.g., `971c7acb99decde45bf28860e6e10372718c51e2`)
3. **Git blame** -- per-line attribution to the commit that last modified each line

For dependencies (CI-03), each package is additionally identified by:
- Crate name and semver version (e.g., `bevy 0.15.3`)
- SHA-256 checksum from `Cargo.lock` (e.g., `2eaad7fe...`)
- Source registry (`registry+https://github.com/rust-lang/crates.io-index`)

## 4. Versioning Strategy

### 4.1 Software Version

| Component | Value | Source |
|-----------|-------|--------|
| Package version | `0.1.0` | `Cargo.toml` line 3 |
| Versioning scheme | Semantic Versioning (semver) | Major.Minor.Patch |
| Rust edition | 2024 | `Cargo.toml` line 4 |
| MSRV | 1.85 | `Cargo.toml` line 5 |

**Semver policy:**
- **Patch (0.1.x):** Bug fixes, documentation updates, test additions. No API or behavior changes.
- **Minor (0.x.0):** New features, new drug models, new calibration profiles, new emergence systems. Backward-compatible.
- **Major (x.0.0):** Breaking changes to axioms, fundamental constants, layer definitions, or batch simulator API. Requires full regression testing and risk reassessment.

**Current status:** RESONANCE is pre-1.0 (`0.1.0`). Per semver convention, any minor version bump may include breaking changes during pre-1.0 development.

### 4.2 Commit-Based Identification

In the absence of formal release tags, every configuration state is uniquely identified by its Git commit hash. All regulatory documents reference the commit hash in their headers.

| Identification Method | Format | Example |
|----------------------|--------|---------|
| Full commit hash | 40-character SHA-1 | `971c7acb99decde45bf28860e6e10372718c51e2` |
| Short commit hash | 7-character prefix | `971c7ac` |
| Commit message | Type-prefixed description | `docs: honesty pass -- 5 known limitations added` |
| Timestamp | ISO 8601 | `2026-04-02T...` |

### 4.3 Dependency Versioning

Dependencies are versioned through two mechanisms:

1. **`Cargo.toml`** -- declares semver constraints (e.g., `bevy = "0.15"` permits 0.15.x)
2. **`Cargo.lock`** -- pins exact resolved versions (e.g., `bevy = 0.15.3` with checksum)

The `Cargo.lock` file is committed to the repository and is the authoritative record of dependency versions. Any change to `Cargo.lock` is visible in Git history and requires the same review process as source code changes.

## 5. Branch Strategy

### 5.1 Trunk-Based Development

RESONANCE uses trunk-based development on a single `main` branch. There are no long-lived feature branches, release branches, or development branches.

| Aspect | Practice |
|--------|----------|
| Primary branch | `main` |
| Feature branches | Short-lived (merged same day or within sprint) |
| Release branches | None (pre-1.0; no formal releases yet) |
| Hotfix branches | None (fixes committed directly to `main`) |
| Branch protection | Not currently configured (gap for RD-7) |

**Justification:** With a small development team and sprint-based workflow, trunk-based development minimizes merge complexity and ensures all changes are immediately visible. The 3,113-test gate provides regression protection equivalent to branch-based CI gates.

### 5.2 Merge Process

All changes to `main` follow this process:

1. **Development:** Code written following `CLAUDE.md` coding standards
2. **Local verification:** `cargo test` (3,113 tests, 0 failures) + `cargo clippy` (0 warnings)
3. **Review:** Verificador checklist (contract, math, DOD, determinism, perf, tests)
4. **Commit:** Atomic commit with type-prefixed message
5. **Push:** Direct to `main` (trunk-based)

**Gap:** No formal pull request process with CI gates. Changes are verified locally before commit. Formal PR workflow with CI is recommended for RD-7 (Release Package).

## 6. Build Reproducibility

### 6.1 Deterministic Builds

Given identical inputs, RESONANCE builds are deterministic:

| Input | Pinning Mechanism | Verification |
|-------|-------------------|-------------|
| Rust compiler version | MSRV 1.85 in `Cargo.toml`; `rustup override set 1.85.0` | `rustc --version` |
| Dependency versions | `Cargo.lock` (committed) | `cargo build --locked` (fails if Cargo.lock stale) |
| Source code | Git commit hash | `git rev-parse HEAD` |
| Feature flags | Default features in `Cargo.toml` | `cargo build --release` (default features) |
| Build profile | `[profile.dev]` and `[profile.dev.package."*"]` in `Cargo.toml` | Reproducible via `cargo build` |

### 6.2 Build Commands

```bash
# Development build (default features)
cargo build

# Release build (optimized)
cargo build --release

# Locked build (exact Cargo.lock versions, fails on mismatch)
cargo build --release --locked

# Headless simulation (no GPU)
cargo run --bin headless_sim -- --ticks 10000 --scale 8 --out world.ppm

# Batch benchmark
cargo bench --bench batch_benchmark
```

### 6.3 Build Environment Specification

| Component | Required Version | Notes |
|-----------|-----------------|-------|
| Rust toolchain | stable, >= 1.85.0 | MSRV declared in `Cargo.toml` |
| Cargo | Bundled with Rust | Dependency resolution and build |
| Target platforms | macOS (ARM64), Linux (x86_64) | Primary development on macOS |
| GPU | Optional | Required for rendering; not for headless/batch |
| System libraries | Platform-provided | Windowing (winit), audio (platform backend) |

### 6.4 Reproducibility Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| No `rust-toolchain.toml` file | Developer may use different Rust version | MSRV 1.85 declared; `--locked` flag prevents dep drift |
| No containerized build (Docker) | System library versions may differ | Cargo.lock pins Rust dependencies; system lib impact limited to rendering |
| f32 cross-platform variance | Floating-point results may differ across architectures | Determinism verified on same platform; cross-platform testing deferred (H-09) |
| No reproducible build hash | Binary hash differs across builds (Rust does not guarantee bit-identical binaries) | Source-level reproducibility via Git + Cargo.lock is sufficient for Class A |

## 7. Change Control

### 7.1 Change Control Process

Changes to configuration items follow the sprint-based lifecycle defined in RD-1.4 §2:

```
1. Scope Definition (sprint README.md)
   -> What changes, why, what enters/exits scope
2. Implementation (Alquimista)
   -> Code, tests, documentation per CLAUDE.md rules
3. Verification (local)
   -> cargo test (3,113 tests), cargo clippy (0 warnings)
4. Review (Verificador)
   -> Contract, math, DOD, determinism, perf, tests
   -> Verdict: PASS / WARN / BLOCK
5. Commit (atomic, type-prefixed)
6. Sprint Closure
   -> Closure criteria verified (grep-based)
   -> Sprint doc archived to docs/sprints/archive/
```

### 7.2 Change Categories

| Category | Examples | Required Review | CI Gate |
|----------|---------|----------------|---------|
| Source code (CI-01) | New system, bug fix, refactor | Verificador PASS | `cargo test` + `cargo clippy` |
| Dependency change (CI-02, CI-03) | New crate, version bump | Verificador PASS + SOUP update (RD-3.2) | `cargo test` + `cargo audit` |
| Axiom/constant change | Fundamental constant modification | **PROHIBITED** (constitutional constraint) | N/A |
| Coding standard change (CI-07) | New rule or exception | Explicit justification + version control | N/A |
| Map configuration (CI-04) | New map, parameter change | Alquimista review | Visual + headless validation |
| Documentation (CI-06--CI-11) | Architecture, design, regulatory | Observador review | Markdown lint (manual) |
| Test change (CI-12) | New test, test modification | Verificador review | `cargo test` |

### 7.3 Constitutional Constraints

The following items have special change control status per `CLAUDE.md`:

| Item | Status | Change Authority |
|------|--------|-----------------|
| 8 Foundational Axioms | **Inviolable** | No authority can change. Changes are wrong by definition. |
| 4 Fundamental Constants | **Inviolable** (physics: KLEIBER, DISSIPATION) / **Recalibratable** (grid: BANDWIDTH, DENSITY_SCALE) | Grid constants may be recalibrated with justification; physics constants never. |
| 5 Absolute Hard Blocks (HB-1 through HB-5) | **Zero tolerance** | No exceptions. No DEBT justification accepted. |
| 12 Strong Default Hard Blocks (SD-1 through SD-12) | **DEBT-justified exceptions** | Inline `// DEBT: <reason>` required for any violation. |

### 7.4 Emergency Changes

For critical defects discovered outside the sprint cycle:

1. **Identify** -- defect reported or discovered
2. **Triage** -- assess impact on simulation correctness, conservation, determinism
3. **Fix** -- minimal change addressing root cause
4. **Regression test** -- new test covering the defect's root cause
5. **Verify** -- full `cargo test` pass
6. **Commit** -- with `fix:` prefix and defect description
7. **Document** -- update affected sprint doc or create new sprint

## 11. Audit Trail

### 11.1 Git as Audit Trail

Git provides a complete, immutable audit trail for all configuration items:

| Audit Capability | Git Feature | Command |
|------------------|------------|---------|
| Who changed what | Commit author + committer | `git log --format='%H %an %ai %s'` |
| When it changed | Commit timestamp | `git log --since='2026-01-01'` |
| What changed | Diff between commits | `git diff 42834c2..971c7ac` |
| Why it changed | Commit message | `git log --oneline` |
| Line-level attribution | Blame | `git blame src/blueprint/equations/determinism.rs` |
| Full history of a file | Log per file | `git log --follow src/blueprint/equations/pathway_inhibitor.rs` |
| State at any point in time | Checkout | `git checkout <commit>` |
| Tamper evidence | SHA-1 hash chain | Inherent to Git's Merkle tree structure |

### 11.2 Audit Trail Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| Git SHA-1 (not SHA-256) | Theoretical collision risk | Acceptable for Class A; Git is transitioning to SHA-256 |
| No electronic signatures | Commits not cryptographically signed to individuals | GPG signing planned for regulatory baselines |
| Local commits before push | Window between commit and remote availability | Trunk-based development minimizes this window |
| Rebase/amend possible locally | Could alter history before push | Policy: no force-push to `main`; GitHub protection rules planned |
