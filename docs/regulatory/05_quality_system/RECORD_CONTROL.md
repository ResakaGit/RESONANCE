# Record Control Procedure

## 1. Purpose

This procedure defines the controls for identifying, storing, protecting, retrieving, retaining, and disposing of quality records within the RESONANCE Quality Management System, in accordance with ISO 13485:2016 §4.2.5.

Quality records differ from documents: **documents** prescribe what to do (procedures, plans, specifications); **records** provide evidence that activities were performed and results achieved. This procedure governs records.

## 2. Scope

This procedure applies to all quality records generated by the RESONANCE development process, including:
- Test execution results
- Sprint closure evidence
- Benchmark results
- Audit findings
- CAPA records
- Release evidence
- Configuration snapshots

## 3. Quality Records Registry

### 3.1 Record Types

| Record Type | Description | Generated By | Frequency | ISO 13485 Clause |
|-------------|-------------|-------------|-----------|-------------------|
| **Test results** | Output of `cargo test` (pass count, fail count, elapsed time) | `cargo test` execution | Every sprint closure | §7.3.6 (verification), §7.5.2.1 (validation) |
| **Sprint closure evidence** | Checked closure criteria in sprint README.md, including grep command outputs | Sprint closure process | Every sprint closure | §7.3.7 (design transfer) |
| **Benchmark results** | Criterion HTML reports from `cargo bench --bench batch_benchmark` | Benchmark execution | Per release or performance sprint | §7.3.6 (verification) |
| **Grep verification output** | Output of grep commands verifying closure criteria (e.g., "0 results for pattern X") | Sprint closure verification | Every sprint closure | §8.2.4 (internal audit) |
| **CAPA records** | Sprint doc "Problema" section documenting root cause, fix, and verification | CAPA process (RD-5.7) | Per nonconformance | §8.5.2, §8.5.3 |
| **Audit records** | Sprint README checkmarks documenting criteria verification results | Internal audit process (RD-5.5) | Every sprint track closure | §8.2.4 |
| **Release evidence** | Commit SHA, `Cargo.toml` version, test count, dependency snapshot (`Cargo.lock`) | Release process | Per release | §7.3.7 |
| **Bozic validation output** | 10-seed validation binary output (combo > mono confirmed N/10) | `cargo run --release --bin bozic_validation` | Per release | §7.3.6 |
| **Nonconformance records** | Description of defect, disposition, and resolution committed to sprint doc | Nonconforming product process (RD-5.6) | Per nonconformance | §8.3 |
| **Commit history** | Git log entries (author, date, message, SHA, diff) | Git | Continuous | §4.2.5 (record identification) |

### 3.2 Record Format

All quality records in RESONANCE are digital. There are no paper records.

| Format | Content | Tool |
|--------|---------|------|
| Markdown (`.md`) | Sprint docs, closure criteria, audit checklists, CAPA narratives | Text editor + Git |
| Plain text (terminal output) | `cargo test` results, grep verification output | Terminal + Git commit message |
| HTML | Criterion benchmark reports | `cargo bench` (auto-generated) |
| Git objects | Commits, diffs, tags, branches | Git |
| Binary (`.bin`) | Evolved genome snapshots for experiment reproducibility | `assets/evolved/*.bin` |
| RON (`.ron`) | Map configurations | `assets/maps/*.ron` |

## 4. Record Storage

### 4.1 Storage Location

All quality records are stored within the Git repository. There is no external database, cloud storage service, or separate document management system.

| Record Type | Storage Path | Git Tracked |
|-------------|-------------|-------------|
| Sprint closure evidence | `docs/sprints/archive/{TRACK}/*.md` | Yes |
| Active sprint evidence | `docs/sprints/{TRACK}/*.md` | Yes |
| Regulatory documents | `docs/regulatory/**/*.md` | Yes |
| Test results | Embedded in sprint closure docs or commit messages | Yes (commit messages are part of Git history) |
| Benchmark reports | Generated locally; key results captured in sprint docs | Partial (reports are generated artifacts; summaries are committed) |
| Configuration snapshots | `Cargo.toml`, `Cargo.lock` | Yes |
| Genome binaries | `assets/evolved/*.bin` | Yes |
| Git history | `.git/` | Inherent (Git stores its own history) |

### 4.2 Storage Integrity

Git provides integrity guarantees for all stored records:

| Property | Mechanism |
|----------|-----------|
| **Content integrity** | SHA-256 hash of every object (commit, tree, blob). Any modification to any record changes the hash, breaking the chain. |
| **Non-repudiation** | Every commit records author identity, email, and timestamp. |
| **Tamper evidence** | Modifying a historical record requires rewriting Git history (`git rebase --force`), which changes all subsequent commit SHAs --- detectable by any clone holder. |
| **Redundancy** | Repository exists on GitHub (remote) and all local clones. Loss of any single copy does not destroy records. |

### 4.3 Known Limitation: Transient Records

Some quality records are transient by nature and not captured in Git:

| Transient Record | Current Practice | Risk | Mitigation |
|------------------|-----------------|------|------------|
| `cargo test` terminal output | Key metrics (pass count, fail count) noted in sprint closure docs | Full output not preserved | Summary metrics are sufficient for Class A software |
| `cargo bench` HTML reports | Generated locally, not committed | Historical benchmark data may be lost | Criterion stores comparison data locally; key results captured in sprint docs |
| Developer environment details (OS, Rust version) | Not systematically recorded | Reproducibility risk for edge cases | `Cargo.lock` pins dependencies; MSRV 1.85 defines minimum toolchain; deterministic RNG eliminates most environment sensitivity |

These limitations are acceptable for the current research-tool classification (IEC 62304 Class A) but should be addressed if SaMD classification is pursued.

## 5. Record Retrieval

### 5.1 Retrieval Methods

| What to Retrieve | Git Command | Example |
|------------------|-------------|---------|
| Current version of any document | `git show HEAD:<path>` | `git show HEAD:CLAUDE.md` |
| Document at a specific point in time | `git show <commit>:<path>` | `git show 971c7ac:CLAUDE.md` |
| Complete change history of a file | `git log --follow -- <path>` | `git log --follow -- docs/sprints/archive/DECOUPLING_AUDIT/README.md` |
| Exact text changes between versions | `git diff <old>..<new> -- <path>` | `git diff abc123..def456 -- CLAUDE.md` |
| All files at a specific release | `git checkout <tag-or-commit>` | `git checkout v1.0.0` |
| Search for content across history | `git log -S "<search-term>"` | `git log -S "DISSIPATION_SOLID"` |
| Sprint closure at specific date | `git log --after="2026-04-01" --before="2026-04-02" -- docs/sprints/archive/` | Find sprints archived on a specific date |

### 5.2 Retrieval by Record Type

| Record Type | Retrieval Path |
|-------------|---------------|
| Sprint closure evidence | `docs/sprints/archive/{TRACK}/README.md` or `SPRINT_*.md` |
| Test count at a commit | `git stash && git checkout <commit> && cargo test 2>&1 \| tail -1 && git checkout -` (or read sprint doc) |
| Configuration at a commit | `git show <commit>:Cargo.toml` and `git show <commit>:Cargo.lock` |
| Regulatory doc version | `git show <commit>:docs/regulatory/05_quality_system/QUALITY_MANUAL.md` |
| CAPA root cause narrative | Search sprint doc for "Problema" section |
| Benchmark comparison | Local criterion report or sprint doc summary |

## 6. Record Protection

### 6.1 Protection Against Loss

| Protection | Mechanism |
|------------|-----------|
| Remote backup | GitHub hosts the repository. Push-on-commit ensures remote is current. |
| Local redundancy | Every developer clone is a full backup of the entire repository history. |
| Branch protection | Trunk-based development on `main`. No force-push to `main` without explicit justification. |

### 6.2 Protection Against Unauthorized Modification

| Protection | Mechanism |
|------------|-----------|
| Git content addressing | SHA-256 hashes make undetected modification impossible. |
| GitHub access controls | Repository access managed via GitHub permissions. |
| Commit attribution | Every change is attributed to an author with timestamp. |
| Trunk-based workflow | All changes go through `main`. No hidden branches with unreviewed changes. |

### 6.3 Protection Against Inadvertent Destruction

| Protection | Mechanism |
|------------|-----------|
| No deletion policy | Obsolete documents are marked `OBSOLETE` in frontmatter, not deleted. |
| Git append-only model | History is append-only by default. `git push --force` is a destructive operation that is not part of standard workflow. |
| Sprint archive immutability | Archived sprint docs in `docs/sprints/archive/` are treated as immutable historical records. |

## 7. Record Retention

### 7.1 Retention Period

All quality records are retained **indefinitely** in Git history. There is no scheduled record destruction.

| Record Category | Retention Period | Justification |
|-----------------|-----------------|---------------|
| Sprint documentation | Indefinite | Historical reference for design decisions and CAPA investigations |
| Test results (embedded in sprint docs) | Indefinite | Evidence of verification at each milestone |
| Regulatory documents | Indefinite | Required for regulatory history if SaMD is pursued |
| Configuration snapshots (`Cargo.lock`) | Indefinite | Reproducibility of any historical build |
| Git commit history | Indefinite | Complete audit trail |
| Benchmark results (captured in sprint docs) | Indefinite | Performance trending |
| Genome binaries (`assets/evolved/*.bin`) | Indefinite | Experiment reproducibility |

### 7.2 Retention Justification

ISO 13485:2016 §4.2.5 requires retention for "at least the lifetime of the medical device as defined by the organization, but not less than 2 years from the date of product release." Since RESONANCE is not currently a medical device, there is no defined device lifetime. The indefinite retention policy exceeds all regulatory minimums.

Git's storage model makes indefinite retention essentially free --- old records consume negligible incremental storage.

### 7.3 Record Disposition

There is no record disposition (destruction) process. Records are never intentionally destroyed. If a record becomes irrelevant, it remains in Git history but is not referenced in active documentation.

## 8. Record Legibility and Identification

### 8.1 Legibility

All records are stored in plain-text formats (Markdown, TOML, RON, Rust source) that are human-readable without specialized software. Git diffs are displayed in standard unified diff format.

### 8.2 Identification

Each record is identifiable by:

| Identifier | Source | Uniqueness |
|------------|--------|------------|
| File path | Repository directory structure | Unique within a commit |
| Git commit SHA | Git content-addressing | Globally unique (SHA-256) |
| Document ID | YAML frontmatter `document_id` field | Unique within regulatory track |
| Sprint ID | Sprint document title (e.g., `DC-4`) | Unique within track |
| Date | YAML frontmatter or Git commit timestamp | Combined with path, identifies specific version |
